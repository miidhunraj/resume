# Building Claude Code from Source

Step-by-step guide for building the Claude Code CLI from this source snapshot.

## 1. Install Bun

Bun is the runtime and bundler. Version 1.3+ required.

```bash
curl -fsSL https://bun.sh/install | bash
```

Verify:

```bash
bun --version
# Should output 1.3.x or higher
```

## 2. Install dependencies

From the project root:

```bash
bun install
```

This installs ~70 npm packages (~390 total with transitive deps). Takes ~2-3 seconds.

If you see warnings about peer dependencies, those are safe to ignore.

## 3. Build

### Development build (unminified, with sourcemaps)

```bash
bun run build
```

Output:
```
Build succeeded in 0.48s
  dist/cli.js (19853 KB)
  dist/cli.js.map (52779 KB)
```

### Production build (minified, no sourcemaps)

```bash
bun run build -- --minify
```

Output:
```
Build succeeded in 0.22s
  dist/cli.js (10118 KB)
```

### Custom version string

```bash
CLAUDE_CODE_VERSION=1.0.0 bun run build
```

## 4. Run

```bash
# Quick test
bun dist/cli.js --version
# => 0.0.1-dev (Claude Code)

# Also works with Node.js
node dist/cli.js --version

# Run with an API key for actual use
ANTHROPIC_API_KEY=sk-ant-xxx bun dist/cli.js
```

## 5. (Optional) Standalone executable

Compiles the bundle into a single Bun binary with no external dependencies:

```bash
bun run build -- --compile
```

## What the build does

The build script (`build.ts`) does the following:

### Step 1 — Bundle with Bun.build()

Takes the entry point `src/entrypoints/cli.tsx` and bundles the entire source tree into a single `dist/cli.js` file. This is similar to what webpack/esbuild/rollup do, but using Bun's built-in bundler.

### Step 2 — Inline build-time macros

These constants are replaced with literal values at compile time (not available at runtime):

| Macro | Default | Env var override |
|-------|---------|-----------------|
| `MACRO.VERSION` | `0.0.1-dev` | `CLAUDE_CODE_VERSION` |
| `MACRO.BUILD_TIME` | current ISO timestamp | — |
| `MACRO.PACKAGE_URL` | `@anthropic-ai/claude-code` | `CLAUDE_CODE_PACKAGE_URL` |
| `MACRO.NATIVE_PACKAGE_URL` | (empty) | `CLAUDE_CODE_NATIVE_PACKAGE_URL` |
| `MACRO.FEEDBACK_CHANNEL` | (empty) | `CLAUDE_CODE_FEEDBACK_CHANNEL` |
| `MACRO.ISSUES_EXPLAINER` | (empty) | `CLAUDE_CODE_ISSUES_EXPLAINER` |
| `MACRO.VERSION_CHANGELOG` | (empty) | `CLAUDE_CODE_VERSION_CHANGELOG` |

### Step 3 — Dead code elimination via feature flags

The source uses `feature('FLAG_NAME')` from `bun:bundle` in conditionals:

```typescript
import { feature } from 'bun:bundle'

if (feature('VOICE_MODE')) {
  // This entire block is removed when VOICE_MODE is false
  const voice = require('./voice.js')
}
```

All ~60 flags default to `false` in our build, which means all internal-only features (voice, coordinator, bridge, daemon, computer use, etc.) are completely tree-shaken from the output. This is how Anthropic ships different builds from one codebase.

### Step 4 — Resolve stub packages

Some packages don't exist on npm (they're Anthropic-internal). The build plugin intercepts these imports and resolves them to stub implementations in `stubs/`:

| Package | What it is |
|---------|-----------|
| `@ant/claude-for-chrome-mcp` | Chrome extension MCP bridge |
| `@ant/computer-use-input` | Computer use input handling |
| `@ant/computer-use-mcp` | Computer use MCP server |
| `@ant/computer-use-swift` | macOS computer use via Swift |
| `@anthropic-ai/sandbox-runtime` | Seccomp/sandbox isolation |
| `color-diff-napi` | Native syntax highlighting |

### Step 5 — Mark external modules

Some packages are only needed at runtime in specific configurations and are kept external (not bundled). If they're not installed, the code gracefully handles their absence since they're behind dynamic `import()` calls:

- Native addons: `audio-capture.node`, `modifiers-napi`
- Cloud providers: `@azure/identity`, `@aws-sdk/client-bedrock`, `@aws-sdk/client-sts`, `@anthropic-ai/foundry-sdk`
- OpenTelemetry exporters (gRPC, HTTP, proto variants)

### Step 6 — Add shebang

The build script prepends `#!/usr/bin/env node` so the output file is directly executable.

## Project structure after build

```
claudecode/
├── build.ts              # Build script (Bun.build() orchestrator)
├── package.json          # Dependencies
├── tsconfig.json         # TypeScript config (src/ path alias)
├── globals.d.ts          # Type declarations for MACRO.* and bun:bundle
├── bun.lock              # Lock file (auto-generated by bun install)
├── node_modules/         # Installed dependencies
├── stubs/                # Stub implementations for internal packages
│   ├── @ant/             # Anthropic-internal Chrome/computer-use stubs
│   ├── @anthropic-ai/    # sandbox-runtime stub
│   ├── color-diff-napi/  # Native syntax highlighting stub
│   └── _empty.ts         # Generic empty stub
├── dist/                 # Build output
│   ├── cli.js            # The bundled CLI (~10-20MB)
│   └── cli.js.map        # Sourcemap (dev builds only)
└── src/                  # Source code
    ├── entrypoints/      # Entry points (cli.tsx, mcp.ts, sdk/)
    ├── main.tsx          # Main CLI orchestrator (~4.7K lines)
    ├── QueryEngine.ts    # Agent conversation loop
    ├── query.ts          # Single API call cycle
    ├── Tool.ts           # Tool type definitions
    ├── tools.ts          # Tool registry
    ├── tools/            # Individual tool implementations
    ├── commands.ts       # Slash command registry
    ├── commands/         # Slash command implementations
    ├── components/       # React (Ink) UI components
    ├── services/         # API, MCP, analytics, compaction
    ├── state/            # AppState store
    ├── hooks/            # React hooks (permissions, suggestions)

    ├── bridge/           # Remote session bridging
    ├── tasks/            # Background task management
    └── utils/            # Utilities (~100+ files)
```

## Troubleshooting

### `Cannot find module 'X'` at runtime

The module is marked as `external` in the build but isn't installed. Either:
- Install it: `bun add X`
- Or ignore it — external modules are optional and only loaded when their feature is used.

### Build error: `No matching export in "stubs/..."`

A stub file is missing a named export that the source imports. Open the importing file, check what it needs, and add the export to the corresponding stub in `stubs/`.

### `feature() must stay inline`

Comments in the source warn about this. The `feature('FLAG')` call must be directly in an `if` condition — not stored in a variable — for Bun's bundler to perform dead code elimination.

### Runtime crash in `parseFrontmatter`

This happens when `.md` files (skill definitions) can't be loaded as strings. The stubs return empty strings. If you need a specific skill to work, create the actual `.md` file with proper frontmatter content.

### `--help` crashes with Commander.js error

The source uses `-d2e` as a short flag, which isn't standard Commander.js syntax. This may require a patched Commander.js version from Anthropic's internal registry. `--version` and `--print` mode work fine.

## How Anthropic's official build differs

The published `@anthropic-ai/claude-code` package on npm is built from the same source with:

1. **Real internal packages** instead of stubs (`@ant/*`, sandbox-runtime, color-diff-napi)
2. **Feature flags selectively enabled** (e.g., `PROACTIVE` is true in the official build)
3. **Vendored native binaries** — `ripgrep` and `audio-capture.node` for each platform (darwin/linux/win32, arm64/x64)
4. **React Compiler** pre-processing step (already baked into many `.tsx` files in this snapshot)
5. **Full minification** producing a ~13MB `cli.js`
6. **Version and changelog** injected from CI/CD pipeline
7. **A patched Commander.js** or custom option parsing for non-standard flags
